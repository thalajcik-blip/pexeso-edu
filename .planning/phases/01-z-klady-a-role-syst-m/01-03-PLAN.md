## Context

Four independent bugs, all blocking production stability:

1. **`addXP` (TECH-02):** `authStore.ts:308–317` does client-side read → increment → write. Under concurrent load (30 students finishing game simultaneously) this produces lost updates. Fix: replace with `add_xp(user_id, delta)` SQL function using `UPDATE ... SET xp = xp + delta` which is inherently atomic.

2. **`game_start` payload (TECH-01):** Two broadcast call sites in `gameStore.ts` (lines ~775 and ~948) send full `CardData[]` objects. Measured payload: 27–29 KB against a 32 KB Supabase Realtime limit. Fix: send only `cardIds: number[]`; each receiver reconstructs cards from the deck already loaded in lobby.

3. **`rooms` RLS (TECH-03):** Current policy is `USING (true)` — any authenticated user can read/write any room. The `host_id` column is `text` (not UUID) and stores a random string from `getPlayerId()`, not `auth.uid()`. Fix requires changing `createRoomInDb` to store `auth.uid()` as `host_id`.

4. **`crypto.getRandomValues()` (TECH-04):** Two sites use `Math.random()` for security-relevant codes. `getPlayerId()` in `multiplayerService.ts:66` also uses `Math.random()` — fix it as part of this task.

### Task 1: Atomic `add_xp()` SQL function and `authStore.ts` call site update

**Files:**
- `supabase/migrations/$(date +%Y%m%d%H%M%S)_add_xp_rpc.sql`
- `src/store/authStore.ts`

**What:** Create atomic SQL function; replace client-side read-modify-write in `authStore.ts`.

**How — migration file:**

```sql
CREATE OR REPLACE FUNCTION add_xp(p_user_id uuid, p_xp_delta integer)
RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  v_new_xp    integer;
  v_new_level integer;
  v_level_xp  integer[] := ARRAY[0, 100, 250, 500, 1000, 2000, 3500, 5000, 7500, 10000];
  i           integer;
BEGIN
  UPDATE profiles
    SET xp = xp + p_xp_delta
  WHERE id = p_user_id
  RETURNING xp INTO v_new_xp;

  -- Compute level from new XP
  v_new_level := 1;
  FOR i IN 1..array_length(v_level_xp, 1) LOOP
    IF v_new_xp >= v_level_xp[i] THEN v_new_level := i; END IF;
  END LOOP;
  v_new_level := LEAST(v_new_level, array_length(v_level_xp, 1));

  UPDATE profiles SET level = v_new_level WHERE id = p_user_id;

  RETURN jsonb_build_object('xp', v_new_xp, 'level', v_new_level);
END;
$$;
```

Apply via `supabase db push`.

**How — `authStore.ts` (lines 308–317):**

Find the `addXP` action. It currently reads `profile.xp`, adds the delta, and writes back. Replace the entire body:

```typescript
addXP: async (amount) => {
  const { user } = get()
  if (!user) return
  const { data, error } = await supabase.rpc('add_xp', {
    p_user_id: user.id,
    p_xp_delta: amount,
  })
  if (!error && data) {
    set(s => ({
      profile: s.profile
        ? { ...s.profile, xp: data.xp, level: data.level }
        : null,
    }))
  }
},
```

The `addXP: (amount: number) => Promise<void>` interface signature stays the same. No changes needed to `gameService.ts:56` — `await useAuthStore.getState().addXP(xpEarned)` continues to work.

**What:** Change `game_start` broadcast type from `cards: CardData[]` to `cardIds: number[]`. Update two broadcast call sites in `gameStore.ts` and the reconstruction logic in `_applyGameStart`.

**How — `multiplayerService.ts` line 46:**


**How — `gameStore.ts` line ~948 `startOnlineGame()`:**

Find the `broadcastGameAction({ type: 'game_start', cards, ... })` call. The `cards` variable is the shuffled `CardData[]` array. Replace:
```typescript
// Before:
broadcastGameAction({ type: 'game_start', cards, playerIds, playerNames, deckId: selectedDeckId, size: selectedSize, turnTime, quizTime, startingPlayer })

// After:
const cardIds = cards.map(c => c.id)
broadcastGameAction({ type: 'game_start', cardIds, playerIds, playerNames, deckId: selectedDeckId, size: selectedSize, turnTime, quizTime, startingPlayer })
```

**How — `gameStore.ts` line ~775 `playAgain()`:**

Same change as above. Find the second `broadcastGameAction({ type: 'game_start', cards, ... })` call and apply the identical transformation.

**How — `gameStore.ts` `_applyGameStart()` (line ~622) and `case 'game_start':` handler (line ~687):**

The receiver now gets `cardIds: number[]` instead of `cards: CardData[]`. It must reconstruct the card array from the local deck.

In the `case 'game_start':` handler (or wherever `_applyGameStart` is called), add a pre-fetch step before reconstruction:

```typescript
case 'game_start': {
  const { cardIds, deckId, ... } = action

  // Ensure local deck is loaded before reconstruction
  let localCards: CardData[] = get().cards  // cards pre-loaded during lobby for this deck

  // For custom decks, fetch if not already loaded
  if (deckId && localCards.length === 0) {
    // fetch the custom deck cards and set into state
    // use existing fetchCustomDeckFull() pattern from the codebase
    const deckCards = await fetchCustomDeckFull(deckId)
    localCards = buildCardsFromDeck(deckCards)
  }

  // Reconstruct ordered card array from cardIds
  // cardIds are indices into the local deck pool — use them to produce the same order as host
  const cardMap = new Map(localCards.map(c => [c.id, c]))
  const orderedCards = cardIds.map(id => cardMap.get(id)).filter(Boolean) as CardData[]

  get()._applyGameStart({ ...action, cards: orderedCards })
  break
}
```

Note: `_applyGameStart` internally still expects `cards: CardData[]` — update its signature to accept the reconstructed array, or keep the internal conversion above and pass it in. Do not change the `_applyGameStart` internal logic beyond accepting the reconstructed `cards`.

If card IDs in `CardData` are integers matching the index in the deck array, the `cardMap` lookup works directly. If `CardData.id` is not guaranteed to be set, use position index as the ID — confirm by checking the existing card-building logic (search for where `CardData` objects are constructed in `gameStore.ts` or `src/data/decks.ts`).

### Task 3: `rooms` RLS — restrict to host + authenticated readers; fix invite/room code entropy

**Files:**
- `supabase/migrations/$(date +%Y%m%d%H%M%S)_rooms_rls_and_crypto.sql`
- `src/services/multiplayerService.ts`
- `src/admin/DeckEditor.tsx`

**What:** Fix `rooms` RLS to restrict UPDATE/DELETE to the room host; restrict SELECT to authenticated users. Store `auth.uid()` as `host_id` for RLS to work. Replace `Math.random()` with `crypto.getRandomValues()` at all three sites.

**How — `multiplayerService.ts` `createRoomInDb()` (line ~72):**

The function currently passes `getPlayerId()` (random string) as `host_id`. Change it to use the authenticated user's UUID:

```typescript
export async function createRoomInDb(roomId: string, settings: RoomSettings): Promise<void> {
  const { data: { user } } = await supabase.auth.getUser()
  const hostId = user?.id ?? getPlayerId()  // fallback to random string if unauthenticated (should not happen)

  await supabase.from('rooms').insert({
    id: roomId,
    host_id: hostId,
    settings,
  })
}
```

Note: `host_id` column is `text` — `user.id` is a UUID string, which is valid as text. No type change needed on the column.

**How — migration file (rooms RLS):**

```sql
-- Drop the all-access open policy
DROP POLICY IF EXISTS "open" ON rooms;

-- Host (authenticated, auth.uid() stored as host_id) manages their room
CREATE POLICY "host_manages_room" ON rooms
  FOR ALL USING (host_id = auth.uid()::text)
  WITH CHECK (host_id = auth.uid()::text);

-- Any authenticated user can SELECT rooms (needed to join via fetchRoomFromDb)
CREATE POLICY "authenticated_reads_rooms" ON rooms
  FOR SELECT TO authenticated USING (true);
```

**How — fix `crypto.getRandomValues()` in `multiplayerService.ts`:**

**Site 1 — `generateRoomCode()` (line ~58–61):**
```typescript
export function generateRoomCode(): string {
  const chars = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789'
  const bytes = new Uint8Array(6)
  crypto.getRandomValues(bytes)
  return Array.from(bytes, b => chars[b % chars.length]).join('')
}
```

**Site 2 — `getPlayerId()` (line ~66):**
```typescript
export function getPlayerId(): string {
  // Before: Math.random().toString(36).substring(2, 10)
  const bytes = new Uint8Array(8)
  crypto.getRandomValues(bytes)
  return Array.from(bytes, b => b.toString(16).padStart(2, '0')).join('')
}
```

**How — fix `crypto.getRandomValues()` in `DeckEditor.tsx` (line ~78–79):**

Find `generateCode()` in `src/admin/DeckEditor.tsx`:
```typescript
function generateCode(): string {
  const bytes = new Uint8Array(4)
  crypto.getRandomValues(bytes)
  const chars = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789'
  return Array.from(bytes, b => chars[b % chars.length]).join('')
}
```

`crypto.getRandomValues` is a global in all modern browsers and Deno — no import needed.
